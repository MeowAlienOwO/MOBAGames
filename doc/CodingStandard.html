1 Introduction:

1.1 The reason to have coding standard

Having coding standard is aiming to:

  • To reduce the cost of switching different type and explain
  • Make the code much more readable, and help others understand the code
  • For convenience of possible further maintainance

1.2 Different levels of coding standard restriction

There are three levels of standard restriction.

  1. NECESSARY :: Every should follow the rule, otherwise the code shouldn't be accepted in the program.
  2. SUGGESTED :: Not necessary, but it's better to follow the rule rather than not follow it.
  3. NO RESTRICTION :: There is no restriction, coders can follow their own style.

In the remaining part of this document, we use those notations to highlight the level of standard restriction:

  • NECESSARY
  • SUGGESTED
  • NO RESTRICTION

1.3 Covered areas

This file will cover the restrictions on following areas:

  • File and file organisation
  • Indentation
  • Comments
  • Declaration
  • Naming Convention
  • Operators, statements and other related things.

2 File and file organisation

2.1 File type

The file in this project includes:

*.java file
java source file.
*.class file
java binary file (shouldn't be tracked by github)
*.org file
(not nessary) emacs org file, for documenting.
*.pdf file
pdf file, for documenting.
Makefile
(not necessary) used to build the whole program and test
README
Introduction file, using markdown notations.
LICENCE
licence for the project.
.gitignore
git ignore file is used to store the rules of untrack files for github.

2.2 Git related rules :: NECESSARY

  • All temp file shouldn't be tracked by github.
  • The gh-source and gh-page branch is used to maintain the project web page.
  • The master branch should be the stable version of codes

2.3 File name rule :: NECESSARY

*.java
should be named by the name of the most important class name in this file
.pdf/.org
these file name should be clear enough to show the main usage of that document

2.4 Source code file organization

2.4.1 package rule :: NECESSARY

  • The first uncommented line should be package information.
  • package name should be all lower-case characters

2.4.2 import rule :: SUGGESTED

  • The import package should put all java package first, then the third-party package

2.4.3 class rule

  1. Order :: NECESSARY

    One certan class should organize its contents according to following order:

    1. static > non-static
    2. variable > constructor > methods
    3. public > protected > default > private
    4. nest class should be put at the end of all methods, if has any.
    5. main method should always be first place
  2. Blank line :: NO RESTRICTION

    Using blank lines to show the seperation.

3 Indentation

3.1 1 tab = 4 space :: NECESSARY

3.2 Use space for better indentation :: SUGGESTED

Space can helping indenting like: 

public    int i = 0;
protected int j = 0;

3.3 Line length :: SUGGESTED

A single line shouldn't be more than 80 characters.

3.4 Wrapping lines :: SUGGESTED

When the expression is longer than a single line, it should be well-breaked. Following breaking is suggested:

  • Breaking after a comma
  • Breaking before a operater

the new line should be well-indented to make the code clean.
